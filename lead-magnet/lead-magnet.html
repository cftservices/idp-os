Build the Missing AI Data Layer — 7 Steps

The 7 Steps

1
Step 1

CONNECT — Ingesting data

Connect to the data source of the PLC or SCADA. OPC-UA is the industry standard: every modern system speaks it. MonsterMQ acts as OPC-UA client and MQTT broker in a single container — no separate bridge needed.

OPC-UA (asyncua) MonsterMQ MQTT Modbus (future)
2
Step 2

CONDITION — Making data reliable

Remove duplicate values, filter noise, convert engineering units. A PLC publishes once per second — but not every value changes. Storing only changed values saves 60–80% of storage space.

MonsterMQ flow engine N8N validation
3
Step 3 — CRITICAL POINT

MODEL — The Semantic Layer (DATA → INFORMATION)

This is where data becomes information. Step 3 creates the semantic layer — the ISA-95 ontology that gives every sensor value its context and meaning. Without this layer, all your data collection is just expensive noise storage.

ISA-95 is not just a hierarchy — it is the complete formal ontology for manufacturing. It defines every entity (Equipment, Material, Personnel, Orders), every relationship (assembled from, defined by, references), and every property (engineering range, alarm limits, asset ID).

Without semantic layer: TIC-001 = 67.3
With semantic layer: techflow/site1/packaging/line1/plc01/temperature
= 67.3°C · Normal: 55–75°C · Asset: WC-PAST-01

The ISA-95 equipment hierarchy maps directly to your MQTT topic structure — every topic level carries context. A JSON asset model is your instance of the ISA-95 ontology: it describes every machine's class, ID, ranges, and relationships.

ISA-95 resource types: Equipment (machines) · Material (inputs/outputs) · Personnel (operators) · Physical Assets (spare parts). All linked through the ontology.

ISA-95 ontology Semantic layer UNS (Unified Namespace) Equipment hierarchy JSON asset model
4
Step 4

STORE — Storing information

MongoDB stores all MQTT messages in the plc_data collection with automatic timestamps. Retention 30 days. No InfluxDB needed — MongoDB handles mixed data (sensor values, batch records, alarms, shift logs) in a single collection.

MongoDB MonsterMQ archiving 30-day retention

5
Step 5

ORCHESTRATE — Making information flow

Every manufacturing process follows the same ISA-95 pattern: Definition → Demand → Result. A recipe defines how a product is made. A production order creates demand. The actual records what really happened.

Definition → Recipe + Bill of Materials (what to make)
Demand → Production Order / Schedule (when to make it)
Result → Actual data (what was really produced)

N8N automates the ISA-95 MOM activities: dispatching production orders (wf-001), collecting material scans (wf-003/004), generating shift reports (wf-005), and detecting faults (wf-006). Six workflows cover a complete packaging line — no MES license required.

N8N workflows ISA-95 Definition→Demand→Result MonsterMQ webhooks Teams / email
6
Step 6

VISUALIZE — Making insights visible

Grafana for time-series analysis and ad-hoc queries (via FastAPI JSON datasource). Next.js for live dashboards that refresh every 5 seconds — ideal for operators on the shop floor. Grafana template variables: one dashboard for 50 machines.

Grafana Next.js FastAPI
7
Step 7

DISTRIBUTE — Enabling action

The Unified Namespace (UNS) makes data available to every system that needs it — ERP, MES, cloud, edge. MonsterMQ on a Raspberry Pi forwards local data to the VPS. REST API for external integrations. MQTT retain flag for startup states.

UNS / MQTT REST API Raspberry Pi edge
